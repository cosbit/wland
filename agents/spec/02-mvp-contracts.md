# High-level contracts

## Desired State

Desired state represents deterministic, user-owned configuration.

It answers:

> What should the system be configured to do?

Desired state is persisted by WLANd and should be safe to diff, validate, render, back up, restore, and apply idempotently.

Desired state should include:

- `metadata`
    - System name, description, version, labels, and annotations.
- `wan`
    - Upstream routed interfaces.
    - DHCP/static addressing.
    - NAT intent.
    - WAN firewall profile.
- `lan`
    - Local network segments.
    - Bridges.
    - Subnets.
    - DHCP ranges.
    - DNS behavior.
    - Optional VLAN IDs.
- `phy`
    - Desired radio configuration.
    - Country code.
    - Band.
    - Channel.
    - Channel width.
    - TX power.
    - Radio policy.
- `bss`
    - SSID.
    - Security mode.
    - Secret reference.
    - LAN assignment.
    - Client policy.
- `wlan`
    - Concrete AP interface binding.
    - PHY binding.
    - BSS binding.
    - LAN binding.
- `firewall`
    - zones
    - zone types
    - router-local service access
    - inter-zone forwarding policy
    - NAT policy
    - explicit port forwards
    - optional advanced escape hatches
- `management`
    - UI/API/SSH access behavior.
    - Management listen scope.
    - Management ports.
- `secrets`
    - References to secrets, not raw secret values.
- `apply_policy`
    - Rollback behavior.
    - Lockout protection.
    - Validation requirements.

Desired state should not include:

- Connected clients.
- DHCP leases.
- Current signal strength.
- Runtime counters.
- Last logs.
- Service PIDs.
- Temporary interface state.
- Kernel-generated state.

---

## Observed State

Observed state represents the current runtime condition of WLANd-managed infrastructure.

It answers:

> Did the system successfully realize the desired network configuration?

Observed state is read-only from the user's perspective. It is generated by WLANd by inspecting Linux interfaces, routes, hostapd process state, nftables state, DHCP/DNS service state, bridges, PHYs, and controller-managed services.

Observed state should include:

- `interfaces`
    - Link state.
    - Carrier state.
    - MAC address.
    - Assigned addresses.
    - Bridge membership.
    - Default route presence.

- `routes`
    - Active routing table.
    - Default route.
    - Per-interface route state.

- `phy`
    - Detected radios.
    - Driver.
    - Supported bands.
    - Supported channels.
    - Supported AP mode.
    - Supported multi-BSS behavior.

- `wlan`
    - Whether the AP interface exists.
    - Whether the AP interface is up.
    - Bound PHY.
    - Bound BSS.
    - Bound bridge.

- `bss`
    - Running/stopped state.
    - SSID.
    - Bound WLAN interface.
    - Bound LAN.
    - Channel.
    - Security mode.
    - hostapd runtime state.

- `services`
    - hostapd state.
    - DHCP service state.
    - DNS service state.
    - nftables state.
    - controller state.

- `firewall`
    - Active zones backend.

- `logs`
    - Recent service logs.
    - Last apply log.
    - Last validation failure.

- `drift`
    - Differences between desired state and infrastructure runtime state.

Observed state should not include:

- Connected clients.
- Associated stations.
- DHCP lease inventory.
- Client hostnames.
- Client signal strength.
- Client traffic counters.
- Per-client authentication state.

Those belong to client runtime state, not observed infrastructure state.

## Client Runtime State

Client runtime state represents devices currently interacting with WLANd-managed networks.

It answers:

> Who is connected or recently visible on the network?

Client runtime state is operational telemetry, not configuration and not infrastructure state.

Client runtime state may be derived from:

- hostapd station tables.
- DHCP lease files.
- ARP/neighbor tables.
- bridge forwarding databases.
- optional mDNS/hostname discovery.
- optional traffic accounting.

Client runtime state should include:

- Associated Wi-Fi stations.
- Recently disconnected stations.
- DHCP leases.
- IP-to-MAC mappings.
- Hostnames, if known.
- BSS association.
- WLAN interface.
- LAN segment.
- Signal strength.
- RX/TX counters.
- Association duration.
- Authentication state.

Client runtime state should not affect desired state unless the user creates an explicit policy from it.

Example:

```json
{
  "timestamp": "2026-05-24T00:00:00Z",
  "stations": {
    "de:ad:be:ef:00:01": {
```
